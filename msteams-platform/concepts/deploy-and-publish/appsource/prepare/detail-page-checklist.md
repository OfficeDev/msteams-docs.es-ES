# <a name="create-a-store-listing-for-your-microsoft-teams-app"></a>Crear una descripción de la tienda para tu Microsoft Teams aplicación

La información que envías al Centro de partners [&#8212;](https://partner.microsoft.com) incluyendo tu nombre, descripciones, iconos e imágenes&#8212;se convierte en la tienda Microsoft Teams y la descripción de Microsoft AppSource para la aplicación.

Una descripción de la tienda puede ser la primera impresión de alguien de la aplicación. Aumente las instalaciones con una descripción que transmita eficazmente las ventajas, funcionalidad y marca de la aplicación.

## <a name="specify-a-short-name"></a>Especificar un nombre corto

El nombre de la aplicación (específicamente, su [*nombre*](~/resources/schema/manifest-schema.md#name)corto) desempeña un papel fundamental en la forma en que los usuarios lo descubren en la tienda.

En el ejemplo siguiente se resalta dónde se muestra el nombre corto de una aplicación en una descripción de la tienda.

:::image type="content" source="../../../../assets/images/store-detail-page/AppName-02.png" alt-text="La captura de pantalla de ejemplo resalta dónde se muestra el nombre corto de una aplicación en una descripción de la tienda.":::

### <a name="best-practices-for-names"></a>Procedimientos recomendados para nombres

**Sí:**

* Elige un nombre sencillo y memorable que insinúa lo que hace la aplicación.
* Ser distintivo.
* Evite errores ortográficos y gramaticales.

**No:**

* Use términos profanos o despectivos.
* Use un lenguaje racial o culturalmente insensible.
* Usa términos genéricos o nombres similares a las aplicaciones existentes.
* Incluya "Teams", "Microsoft", nombres de producto de Microsoft existentes o próximos, o "aplicación" en el nombre.

> [!NOTE]
> Si la aplicación forma parte de una asociación oficial con Microsoft, el nombre de la aplicación debe ser el primero (por ejemplo, *Salesforce Connector para Microsoft Teams*).

## <a name="write-descriptions"></a>Escribir descripciones

Necesitas una descripción corta y larga de la aplicación.

### <a name="short-description"></a>La descripción breve

Un resumen conciso de la aplicación que debe ser original, atractivo y dirigido a la audiencia de destino. Lo ideal es mantener la descripción corta en una oración.

En el ejemplo siguiente se resalta dónde se muestra la descripción breve de una aplicación en una descripción de la tienda:

:::image type="content" source="~/assets/images/store-detail-page/ShortDescription-02.png" alt-text="La captura de pantalla de ejemplo resalta dónde se muestra la breve descripción de una aplicación en una descripción de la tienda.":::

#### <a name="best-practices-for-short-descriptions"></a>Procedimientos recomendados para descripciones breves

**Sí:**

* Coloque primero la información más importante.
* Incluir palabras clave que es probable que los clientes busquen.

**No:**

* Repita el nombre de la aplicación.
* Dependa de la jerga o terminología especializada. (No puede asumir que los usuarios saben qué buscar).

### <a name="long-description"></a>La descripción larga

La descripción larga puede proporcionar una narrativa atractiva que resalta las características principales de la aplicación, los problemas que resuelve y su audiencia de destino. Aunque esta descripción puede tener hasta 4000 caracteres, la mayoría de los usuarios solo leerán entre 300 y 500 palabras.

En el ejemplo siguiente se resalta dónde se muestra la descripción larga de una aplicación en una descripción de la tienda:

:::image type="content" source="~/assets/images/store-detail-page/ShortDescription-02.png" alt-text="Destaca la captura de pantalla de ejemplo donde se muestra la descripción larga de una aplicación en una descripción de la tienda.":::

#### <a name="usage-examples"></a>Ejemplos de uso

Las siguientes frases son ejemplos de lo que se permite al escribir descripciones largas:

* "<El *nombre de la* aplicación> funciona con Microsoft Teams"
* "... un <*de aplicación>* para Microsoft Teams"
* "<el *nombre de la* aplicación> se integra con Microsoft Teams"
* "... integrado con Microsoft Teams"
* "... para usuarios que trabajan con Microsoft Teams"
* "... para <*tareas específicas>* dentro de Microsoft Teams"
* "... basado en ..."
* "... se ejecuta en ..."
* "... habilitado por ..."
* "... desarrollado para ..."
* "... diseñado para ..."

#### <a name="best-practices-for-long-descriptions"></a>Procedimientos recomendados para descripciones largas

**Sí:**

* Use [Markdown para](https://support.office.com/article/use-markdown-formatting-in-teams-4d10bd65-55e2-4b2d-a1f3-2bebdcd2c772) dar formato a la descripción.
* Enumera las características con puntos de viñeta para que sea más fácil examinar la descripción.
* Use voz activa y hable directamente con los usuarios (por ejemplo, *Puede ...*).
* Incluye un vínculo de ayuda o soporte técnico.
* Identifique lo siguiente si procede: limitaciones, configurar información, dependencias de cuenta y actualizaciones de versiones.

**No:**

* Supere las 500 palabras.
* Incluir demasiadas palabras clave. (Es una distracción y no ayuda a los usuarios a encontrar la aplicación).
* Usa el siguiente idioma a menos que la aplicación haya pasado por un proceso de certificación oficial:
  * "... certificado para ..."
  * " ... powered by ..."

### <a name="best-practices-for-all-descriptions"></a>Procedimientos recomendados para todas las descripciones

**Sí:**

* Haga referencia a nombres de productos de Microsoft solo cuando sea necesario. Para obtener más información sobre las directrices, consulte [Directrices de marca y marca de Microsoft](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/usage/general).
* Si necesita hacer referencia a **Teams**, escriba la primera referencia como **Microsoft Teams**. Las referencias posteriores se pueden acortar **a Teams**.
* Consulte **Microsoft 365** en lugar de **Office 365**.
* Evite errores ortográficos y gramaticales.
* Evite mayúsculas innecesarias (por ejemplo, **Usuarios** en lugar de **usuarios**).
* Evite los acrónimos.

**No:**

* Abreviar Microsoft como **MS** o **MSFT**.
* Indica que la aplicación es una oferta de Microsoft, incluido el uso de lemas o lemas de Microsoft.
* Usa nombres de marca con derechos de autor que no posees.

## <a name="adhere-to-icon-design-guidelines"></a>Cumplir las directrices de diseño de iconos

Los iconos son uno de los elementos principales que los usuarios ven al navegar por la tienda. Los iconos deben comunicar el propósito de la marca de la aplicación a la vez que se adhieren a Teams requisitos.

Para obtener más información, [consulta instrucciones específicas sobre cómo diseñar Teams iconos de la aplicación](~/concepts/build-and-test/apps-package.md#app-icons).

## <a name="capture-screenshots"></a>Capturar capturas de pantalla

Las capturas de pantalla proporcionan una vista previa visual destacada de la aplicación para complementar el nombre, el icono y las descripciones de la aplicación.

### <a name="requirements-for-screenshots"></a>Requisitos para capturas de pantalla

* Hasta cinco capturas de pantalla por descripción.
* Los tipos de archivo admitidos incluyen PNG, JPEG y GIF.
* Las dimensiones deben ser de 1366 x 768 píxeles.
* Tamaño máximo de 1.024 KB.

### <a name="best-practices-for-screenshots"></a>Procedimientos recomendados para capturas de pantalla

**Sí:**

* Céntrate en las capacidades de la aplicación (por ejemplo, cómo las personas pueden comunicarse con el bot).
* Incluye contenido que represente con precisión la aplicación.
* Use el texto con juicio.
* Capturas de pantalla de fotogramas con un color que refleje su marca e incluya contenido de marketing, similar al siguiente ejemplo de [Freshdesk](https://appsource.microsoft.com/product/office/WA104381505?src=office&tab=Overview) (los requisitos de dimensión se aplican a toda la imagen y no solo a la captura de pantalla): ejemplo de captura de pantalla de :::image type="content" source="../../../../assets/images/freshdesk.png" alt-text="Freshdesk"::: de aplicaciones de terceros

**No:**

* Mostrar dispositivos específicos, como teléfonos o portátiles.
* Muestra cromo o interfaz de usuario que no esté en la aplicación.
* Captura cualquier Teams o interfaz de usuario del explorador en las capturas de pantalla.
* Incluye mockups que reflejen inexactamente la interfaz de usuario real de la aplicación, como mostrar la aplicación en un explorador en lugar de una Teams pestaña.

Para obtener más prácticas recomendadas, consulta [Crear imágenes eficaces para las tiendas de aplicaciones de Microsoft.](/office/dev/store/craft-effective-appsource-store-images)

## <a name="create-a-video"></a>Crear un vídeo

Un vídeo puede ser la forma más eficaz de comunicar por qué las personas deben usar la aplicación. Debe abordar las siguientes preguntas en un vídeo:

* Quién Cuál es tu aplicación?
* ¿Qué problemas puede resolver la aplicación?
* ¿Cómo funciona la aplicación?
* ¿Qué otras ventajas obtienes al usar la aplicación?

Si incluyes un vídeo, aparecerá antes de las capturas de pantalla en la descripción.

### <a name="best-practices-for-videos"></a>Procedimientos recomendados para vídeos

* Mantén el vídeo entre 30 y 90 segundos.
* Apunta a la calidad. En una descripción, los usuarios verán el vídeo antes de las capturas de pantalla.
